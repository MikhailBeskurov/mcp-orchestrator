## Версия 0.8.5 — Автоматизация и надёжность оркестрации

Снижение церемониала MCP-вызовов, защита от незавершённых сессий, консистентность API.

> **Мотивация:** реальная оркестрация v0.8 потребовала ~25 MCP-вызовов на 2-файловую задачу.
> Ручные `file_lock`/`file_unlock`, минимальные ответы `task_create`, невозможность закрыть сессию —
> всё это тормозит работу и создаёт риск orphaned-состояний при обрыве сессии.

| # | Пункт | Статус | Описание |
|---|-------|--------|----------|
| **10.1** | `session_update` | `[x]` | Обновление статуса сессии (`completed` / `abandoned`). При завершении — **автоочистка**: разблокировка всех файлов сессии, перевод `pending`-задач в `failed`. Решает проблему «вечно активных» сессий. |
| **10.2** | Автоблокировка файлов в жизненном цикле задачи | `[x]` | `task_create` с `files` → автоматический `file_lock` за `assigned_to`. `task_update(done/failed)` → автоматический `file_unlock`. Отдельные `file_lock`/`file_unlock` остаются для ручного контроля. Убирает ~4 MCP-вызова на задачу. |
| **10.3** | Консистентные ответы API | `[x]` | `task_create` и `review_submit` возвращают **полный объект** из БД (как `session_start`, `memory_store`), а не минимальный `{id, status}`. Убирает лишние `task_get`-вызовы. |
| **10.4** | Роль `orchestrator` в AGENT | `[x]` | Добавить `orchestrator` в enum `assigned_to` (`task_create`) и `locked_by` (`file_lock`). Сейчас оркестратор может отправлять сообщения, но не может заблокировать файл за собой. Обновить MCP-схему инструментов. |
| **10.5** | Очистка orphaned-состояний при старте | `[x]` | При `session_start` — проверить и очистить: блокировки файлов от `abandoned`-сессий, `pending`/`in_progress`-задачи без активной сессии. Защита от обрыва IDE / краша оркестратора. |

## Версия 0.9.5 — Снижение церемониала оркестрации (раунд 2)

Фидбек из реальной оркестрации v0.9: ~28 MCP-вызовов на 2-файловую задачу. Основные bottleneck'и — обязательный `task_update(in_progress)` сразу после `task_create` и отсутствие батч-операций для ревью.

> **Мотивация:** v0.8.5 убрала ручные `file_lock`/`file_unlock` (~4 вызова), но полный цикл
> с ревью всё ещё требует ~28 вызовов. Батч-операции и начальный статус задачи
> уберут ещё ~10 вызовов, доведя до ~18.

| # | Пункт | Статус | Описание |
|---|-------|--------|----------|
| **11.1** | Начальный статус в `task_create` | `[x]` | Опциональный параметр `status` в `task_create` (default `pending`, допустимо `in_progress`). При `in_progress` — задача сразу стартует, не нужен отдельный `task_update`. Убирает 2–4 вызова за сессию. |
| **11.2** | `review_submit_batch` | `[x]` | Батч-отправка замечаний: принимает массив замечаний, возвращает массив созданных объектов. Один вызов вместо N. Валидация каждого элемента, атомарная транзакция (все или ничего). |
| **11.3** | `review_resolve_batch` | `[x]` | Батч-закрытие замечаний: принимает массив `[{id, status, resolve_comment?}]`, возвращает массив обновлённых объектов. Один вызов вместо N. Валидация существования каждого ID. |
| **11.4** | Обновление правила оркестрации | `[x]` | В `.cursor/rules/orchestration.mdc` шаг 6 — явно добавить `session_update(completed)`. Добавить рекомендацию: ревьюеры возвращают JSON-массив замечаний для батч-загрузки через `review_submit_batch`. |
